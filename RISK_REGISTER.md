# Risk Register

| Risk | Category | Probability | Impact | Severity | How to detect | Mitigation |
|---|---|---|---|---|---|---|
| Custom migration runner diverges from documented Alembic behavior and supports only a linear chain | Release safety / data | High | Critical | P0 | Migration contract failures, deploy startup errors, unknown revision errors | Move toward standard Alembic flow or harden runner with downgrade/rehearsal tests and clearer operator contract |
| SQLite-based test/e2e startup hides PostgreSQL-only migration issues | Migration safety | High | High | P0 | E2E boot warnings, prod-only schema failures | Add Postgres-backed migration and E2E smoke, dialect guards, disable unsafe autoupgrade in SQLite |
| `backend/apps/admin_ui/routers/api.py` contains 115 routes in one file with manual auth/scope patterns | Architecture / RBAC | High | High | P1 | Unauthorized route drift, hard-to-review diffs | Split router by domain and add route-level RBAC smoke matrix |
| `backend/apps/admin_ui/services/candidates.py` and `backend/apps/bot/services.py` are monoliths with side-effect-heavy flows | Architecture / reliability | High | High | P1 | Frequent regressions in unrelated flows, low review confidence | Slice by capability and expand integration tests around seams |
| Recruiter chat rate limiting uses in-memory store, not shared backend | Reliability / scaling | High | High | P0 | Different workers allow inconsistent limits, incident reports, logs | Move to Redis-backed limiter and add multi-worker tests |
| Recruiter chat status model stops at `queued/sent/failed` and does not expose provider acceptance/delivery lifecycle | Reliability / observability | High | High | P0 | Recruiter reports “message sent” but candidate did not receive it | Introduce unified delivery lifecycle and UI diagnostics per message |
| WebApp booking SQL path diverges from domain reservation path | Data integrity | Medium-high | Critical | P0 | Candidate booking conflicts behave differently than admin/bot flows | Refactor to shared domain service or parity-test every path on Postgres |
| Candidate status change and slot creation/scheduling are not fully enclosed in explicit end-to-end transactional boundaries | Data integrity | Medium | Critical | P0 | Duplicate intro days, inconsistent status/slot combinations | Add transactional integration tests and enforce single orchestration seam |
| Background startup wiring in `admin_ui/app.py` and `state.py` has many hidden side effects | Reliability | Medium-high | High | P1 | Boot loops, degraded mode without clear root cause | Introduce startup matrix tests and isolate bootstrapping responsibilities |
| External integrations (Telegram, Max, hh, OpenAI) rely on partial mocks and limited sandbox acceptance | Reliability / release | Medium-high | High | P1 | Staging-only incidents, unobserved failure modes | Add sandbox contract tests and pre-release integration smoke |
| RBAC relies on a mix of router dependencies, guards, and manual service filters | Security | Medium-high | High | P0 | IDOR or recruiter access leakage | Build route matrix tests and consolidate scope application |
| Staff chat attachments and thread membership paths have sparse access-control coverage | Security / privacy | Medium | High | P1 | Foreign attachment/thread access possible | Add attachment/member access matrix and signed URL/download tests |
| Performance gate exists only as manual process, not CI/release enforcement | Performance / release | Medium-high | High | P1 | Regressions discovered after deploy | Add scheduled perf smoke and pre-release gate using existing scripts |
| Existing load results show p95 flapping unless runtime is tuned and noisy features disabled | Performance | Medium | High | P1 | `docs/performance/results_20260301_go_gate.md` regressions | Define canonical perf runtime profile and monitor p95/p99 + error rate |
| No distributed tracing and only partially structured logs across services | Observability | High | Medium-high | P2 | Slow incident investigation, poor correlation | Standardize structured fields and trace propagation, add log contract tests |
| Notification/outbox worker health is visible, but action-level observability is incomplete | Observability / reliability | Medium | High | P1 | Messages disappear into retries without user-facing clarity | Expose outbox/delivery states in admin UI and metrics dashboards |
| Feature flags and env-mode conditionals (`ALLOW_DEV_AUTOADMIN`, `ALLOW_LEGACY_BASIC`, bot toggles, AI toggles) increase behavior variance | Testability / security | Medium-high | Medium-high | P1 | “Works locally, fails in staging/prod” | Build environment matrix tests and explicit prod-safe config assertions |
| Metrics and health endpoints are tested, but alert thresholds and SLO gates are not codified | Observability / operations | Medium | Medium-high | P2 | Silent degradation, no actionable alarms | Define SLO-style thresholds and alert smoke checks |
| Frontend smoke exists, but critical role-based flows and messenger/introduction-day UIs lack deep E2E coverage | UI reliability | Medium-high | High | P1 | Business UI regressions escape to prod | Expand Playwright to role-based critical flows |
| Candidate communication flows may leak or omit PII in logs/audit/export paths | Security / privacy | Medium | High | P1 | Log review, support tickets, audit misses | Add log redaction tests and export/attachment security review |
